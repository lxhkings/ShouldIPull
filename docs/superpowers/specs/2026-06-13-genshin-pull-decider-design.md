## 1. 目标与背景

### 一句话定位
一个**英文**单页网页，帮原神（及后续其他抽卡游戏）玩家在出新角色时回答高频纠结问题：**"Should I pull?"**——直接给结论（🟢 Pull / 🟡 Wait / 🔴 Skip），而不是甩一堆数字让玩家自己愁。

### 为什么做英文站
- **真痛点、高频、全球通用**：原神/星穹/绝区零等每个版本出新角色，玩家都纠结「抽不抽」，英文社区一样天天问。
- **变现强**：美国/英文流量广告 RPM 高出中文几倍，打赏（Ko-fi）也容易。
- **无部署痛点**：Vercel 全球直连，**不用备案、不碰 GFW**——绕开了中文站最大的两个坑。
- **分发适合没受众的人**：靠 Reddit（r/Genshin_Impact、r/HonkaiStarRail、r/gachagaming）+ Google SEO 带量，不用自己拉粉。`"should i pull"` 本身就是高意图搜索词。

### 缺口（差异点）
现有英文工具（paimon.moe 等）都在算「怎么算概率」，几乎没人做「**给结论 + 可分享结果卡**」。Reddit 又特别爱转结果卡片图——这是传播入口。

### 差异化楔子（产品核心赌注）⭐
这是本项目唯一能赢老工具的地方，**资源向这里倾斜**：

- **认清现实**：「计算器+结论」本身没护城河、好抄；老工具 SEO 已扎根，硬拼算法/排名赢不了。
- **唯一胜负手 = 社交传播**：把宝押在「**一张 Reddit 玩家愿意截图斗图的结果卡 + 一句扎心/好笑的结论**」。老计算器恰恰没有这个。
- **结论文案要有「梗」**，用 gacha 圈黑话，能引发共鸣/转发，而不是干巴巴报数字。例：
  - 🔴 `"SKIP — you're 140 wishes short. Pure copium."`
  - 🟡 `"WAIT — 52%. That's a coinflip, anon. Save a bit more."`
  - 🟢 `"PULL — 91%. The gacha gods smile upon you."`
  - （F2P / whale / copium / hopium / anon / pity 这些黑话是 Reddit 通行语，用对了就对味）
- **结果卡设计目标**：竖版、好看、自带 ShouldIPull 水印+网址，让「截图发帖」本身成为获客渠道。卡片质量 = 传播率 = 项目生死。
- **衡量楔子成不成**：分享图被生成/转发的比例，比单纯访问量更能说明「这东西有没有戳中人」。

### 约束：避开商标
"Genshin"/"HoYoverse" 是米哈游商标，**不进产品名和域名**。招牌用功能词（ShouldIPull），页面内文案写 "Genshin Impact banner" 描述用途没问题。

### 本次验证目标
验证「**有没有人用**」。成功信号 = 有人主动使用、并把分享图转发到社区。不追求功能完整，先把最小闭环跑通发出去。

### 明确不做（YAGNI）
- ❌ 账号 / 登录 / 用户系统
- ❌ 实时玩家面板、抽卡记录导入
- ❌ 后端、数据库
- ❌ 专武单独卡池详细建模（v1 只在结论里给一句「要专武则原石差约 N 抽」的粗略提示）
- ❌ 多游戏（星穹等留到验证通过后）
- ❌ 小程序 / App

## 2. 核心功能（MVP 范围）

UI 文案全英文（下表「界面文案」列为实际显示文字）。

### 2.1 输入表单
| 字段 | 界面文案 | 类型 | 说明 |
|---|---|---|---|
| 当前可用抽数 | "Wishes / Primogems you have" | 数字输入 | 能换多少抽（提示 1 wish = 160 primogems） |
| 保底进度 | "Pity (pulls since last 5★)" | 数字输入，0~89 | 距上次出5星垫了多少抽 |
| 歪没歪 | "Guaranteed?" → Yes (next is guaranteed) / No (50/50) | 二选一 | 决定是否处于「大保底」状态 |
| 目标 | "Goal" → Just 1 / Constellations (2~7) | 单选 | 想要的限定数量 |
| 专武 | "Also want the signature weapon" | 复选框 | 勾选则结论加一句专武缺口粗估 |

### 2.2 概率计算（核心）
用**蒙特卡洛模拟**算「到手概率」。原神公开规则：

**规则一·保底机制（单抽出5星概率随垫刀数变化）：**
| 抽数（pity） | 出5星概率 |
|---|---|
| 1~73 | 0.6% |
| 74~89 | 从 0.6% 起每抽约 +6%（软保底） |
| 90 | 100%（硬保底，必出） |

软保底逐抽递增的具体数值采用社区广泛验证的拟合值（paimon.moe 同款模型）：第74抽起每抽概率 ≈ 上一抽 + 6%，到第90抽达到 100%。

**规则二·50/50：**
- 每出一个5星，50% 概率是当期限定，50% 是常驻（歪）。
- 歪了之后，下一个5星 100% 是限定（大保底）。
- 玩家当前「歪没歪」决定模拟起始状态。

**模拟逻辑（伪代码）：**
```
成功次数 = 0
重复 N_SIM (=10000) 次:
    pity = 玩家输入的已垫抽数
    保底状态 = 玩家输入的歪没歪
    剩余抽数 = 玩家输入的抽数
    到手限定数 = 0
    while 剩余抽数 > 0:
        pity += 1; 剩余抽数 -= 1
        if 命中5星(按规则一的 pity 概率):
            if 保底状态 == 大保底 or 掷硬币命中50/50:
                到手限定数 += 1
                保底状态 = 50/50   # 拿到限定后重置
            else:
                保底状态 = 大保底   # 歪了，下次必中
            pity = 0
        if 到手限定数 >= 目标数量: break
    if 到手限定数 >= 目标数量: 成功次数 += 1
到手概率 = 成功次数 / N_SIM
```
10000 次模拟在浏览器里瞬间完成（<50ms）。

**专武粗估（仅当勾选）：** 原神专武池保底机制不同（80抽硬保底、有定轨），v1 不精细建模，只给一句静态估算提示：「要专武的话，按平均约 80 抽，你的原石还差约 X 抽」。X = max(0, 期望总需求 - 当前抽数) 的粗略值。明确标注「专武为粗估，仅供参考」。

### 2.3 结论输出（英文）
模拟完成后展示：
- **决策灯**：🟢 Pull / 🟡 Wait / 🔴 Skip
  - 阈值（v1 简单规则，可调）：到手概率 ≥ 70% → 🟢；40%~70% → 🟡；< 40% → 🔴
- **到手概率**："87% chance to get them" + 进度条
- **一句「梗」结论**（核心，见差异化楔子）：按概率档位 + 输入，从文案库挑一句有梗、对味的话。例：
  - 🔴 "SKIP — you're 140 wishes short. Pure copium."
  - 🟡 "WAIT — 52%. That's a coinflip, anon."
  - 🟢 "PULL — 91%. The gacha gods smile upon you."
- **2~3 条事实理由**（英文，动态生成，给信服力）：
  - "80 wishes + 20 pity + guaranteed → almost certain"
  - "You'll likely have ~10 wishes left"
  - "Not enough for the weapon — short by ~40 wishes"
- **算法依据小字**："Based on official rates + pity system · 10,000 Monte Carlo simulations"

### 2.4 分享功能（项目核心，非附属）
这是差异化楔子的落地，质量决定项目生死，**投入最多打磨**。
- 出结果后显著位置放 **"Share result" / [Save image]** 按钮。
- 渲染一张**竖版结果卡**：🟢🟡🔴 决策灯 + 大号梗结论 + 到手概率进度条 + 关键输入摘要。
- 设计要求：好看、对味、移动端清晰；底部 **ShouldIPull 水印 + shouldipull.com**，让截图发帖=获客。
- 文案库按档位分组（SKIP/WAIT/PULL 各备多条梗文案，随机或按输入选），避免千篇一律。
- 实现：用 `html2canvas`（CDN 引入）把结果卡 DOM 转成图片下载，无需后端。
- （可选增强，时间够再做）"Copy link" 带参数 URL，让别人点开直接看到同样的结果。

## 3. 技术方案

### 选型（小白友好 + Vercel 秒上线）
| 项 | 选择 | 理由 |
|---|---|---|
| 语言 | **英文 UI/文案**（Claude 代写） | 面向全球玩家，变现强，分发靠 Reddit/Google |
| 形态 | 纯前端静态单页 | 无审核、无账号、发链接即可测 |
| 技术栈 | 原生 HTML + CSS + JavaScript，**无构建步骤** | 小白门槛最低，一个文件夹直接部署 |
| 概率计算 | 浏览器内 JS 跑蒙特卡洛 | 无后端，10000 次模拟瞬时完成 |
| 分享图 | `html2canvas`（CDN） | 一个库搞定 DOM 转图，无后端 |
| 部署 | Vercel（全球直连，无需备案） | 连 GitHub 自动部署，免费 HTTPS + 自定义域名 |
| 域名 | 先免费 `shouldipull.vercel.app`，验证后买 `.gg`/`.com` | 0 成本起步 |

### 文件结构（极简）
```
/
├── index.html      # 页面结构 + 表单 + 结果区
├── style.css       # 样式（移动端优先，玩家多在手机看）
├── gacha.js        # 概率模拟 + 结论生成 + 分享图触发
└── docs/...         # 设计文档
```
无 package.json、无构建。Vercel 当作静态站点直接托管。

### 数据流
```
用户填表单 → 点「帮我决定」
  → gacha.js 读取输入
  → 蒙特卡洛模拟 10000 次 → 到手概率
  → 套阈值得决策灯 + 生成理由文案
  → 渲染结果卡片
点「生成分享图」→ html2canvas 截结果卡片 → 下载竖图
```

### 移动端优先
玩家主要在手机上看社区，页面按手机竖屏布局设计，桌面端居中单列即可。

## 4. 错误处理与边界
| 情况 | 处理 |
|---|---|
| 抽数为空/非数字/负数 | 表单校验，提示「请填写有效抽数」，不进入计算 |
| 已垫抽数 > 89 | 限制输入范围 0~89，超出则钳制并提示 |
| 命座目标 > 7 | 钳制到 7（满命） |
| 到手概率 0% 或 100% | 正常展示，理由文案对应「再多攒攒」或「稳了」 |
| html2canvas 加载失败 | 分享按钮降级为提示「截图保存即可」，不阻塞主功能 |

## 5. 测试与验收

### 概率正确性（关键，必须验证）
校验蒙特卡洛结果，方法：用**独立写的精确算法（动态规划/概率递推）**算同一场景，两者误差应在 ±2% 内。不靠手填的经验数字当标准（容易记错），而是两套独立实现互相对照。

可直接断言的边界基准：
| 场景 | 基准 | 验收 |
|---|---|---|
| 从0垫、大保底、90抽、目标1个限定 | ≈ 100%（第90抽硬保底必出，大保底必中限定） | ≈100% |
| 从0垫、50/50、0抽 | 0%（没抽数不可能到手） | 严格0% |
| 已垫89、50/50、1抽、目标1个 | 出5星必中（硬保底），但仅50%是限定 → ≈50% | ≈50% |
| 平均出一个限定所需抽数 | 社区公认 ≈ 93~94 抽 | 单独跑均值校验，落在区间内 |

其余场景（如「从0垫、50/50、90抽」）以 DP 精确解为准交叉验证，不预设固定百分比。

### 功能验收
- [ ] 表单各字段可输入、校验生效
- [ ] 点「帮我决定」出决策灯 + 概率 + 理由
- [ ] 阈值正确映射 🟢🟡🔴
- [ ] 分享图能生成、含工具名+网址、手机上清晰
- [ ] 手机浏览器正常显示与操作
