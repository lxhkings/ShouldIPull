# 抽不抽？· 原神卡池决策器 — 设计文档

- 日期：2026-06-13
- 状态：待用户确认
- 作者：xiaohong + Claude

## 1. 目标与背景

### 一句话定位
一个单页网页，帮原神玩家在出新角色时回答一个高频纠结问题：**「我现在这点家底，到底该不该抽？」**——直接给结论，而不是甩一堆数字让玩家自己愁。

### 为什么做这个
- **真痛点、高频**：原神每个版本出新角色，玩家都会纠结「抽不抽」。
- **缺口明显**：现有工具（paimon.moe 等）都在算「怎么算概率」，几乎没人做「帮你决定抽不抽 + 给结论」。
- **小白可扛**：纯前端、无后端、无账号登录、不碰实时数据，规则全部公开。
- **可被发现**：玩家聚集在 NGA、米游社、B站、小红书、贴吧，能直接贴链接，搜索也能带量。

### 本次验证目标
验证「**有没有人用**」。成功信号 = 有人主动使用、并把分享图转发到社区。不追求功能完整，先把最小闭环跑通发出去。

### 明确不做（YAGNI）
- ❌ 账号 / 登录 / 用户系统
- ❌ 实时玩家面板、抽卡记录导入
- ❌ 后端、数据库
- ❌ 专武单独卡池详细建模（v1 只在结论里给一句「要专武则原石差约 N 抽」的粗略提示）
- ❌ 多游戏（星穹等留到验证通过后）
- ❌ 小程序 / App

## 2. 核心功能（MVP 范围）

### 2.1 输入表单
| 字段 | 类型 | 说明 |
|---|---|---|
| 当前可用抽数 | 数字输入 | 玩家手里能换多少抽（原石/纠缠之缘换算，给个换算提示：1抽=160原石） |
| 保底进度（已垫抽数） | 数字输入，0~89 | 距离上次出5星垫了多少抽 |
| 上把限定歪没歪 | 二选一：歪了(下次必中) / 没歪(50/50) | 决定是否处于「大保底」状态 |
| 目标 | 单选：只要1个 / 要命座(可填2~7) | 想要的限定数量 |
| 顺带要专武 | 复选框 | 勾选则结论里加一句专武缺口粗估 |

### 2.2 概率计算（核心）
用**蒙特卡洛模拟**算「到手概率」。原神公开规则：

**规则一·保底机制（单抽出5星概率随垫刀数变化）：**
| 抽数（pity） | 出5星概率 |
|---|---|
| 1~73 | 0.6% |
| 74~89 | 从 0.6% 起每抽约 +6%（软保底） |
| 90 | 100%（硬保底，必出） |

软保底逐抽递增的具体数值采用社区广泛验证的拟合值（paimon.moe 同款模型）：第74抽起每抽概率 ≈ 上一抽 + 6%，到第90抽达到 100%。

**规则二·50/50：**
- 每出一个5星，50% 概率是当期限定，50% 是常驻（歪）。
- 歪了之后，下一个5星 100% 是限定（大保底）。
- 玩家当前「歪没歪」决定模拟起始状态。

**模拟逻辑（伪代码）：**
```
成功次数 = 0
重复 N_SIM (=10000) 次:
    pity = 玩家输入的已垫抽数
    保底状态 = 玩家输入的歪没歪
    剩余抽数 = 玩家输入的抽数
    到手限定数 = 0
    while 剩余抽数 > 0:
        pity += 1; 剩余抽数 -= 1
        if 命中5星(按规则一的 pity 概率):
            if 保底状态 == 大保底 or 掷硬币命中50/50:
                到手限定数 += 1
                保底状态 = 50/50   # 拿到限定后重置
            else:
                保底状态 = 大保底   # 歪了，下次必中
            pity = 0
        if 到手限定数 >= 目标数量: break
    if 到手限定数 >= 目标数量: 成功次数 += 1
到手概率 = 成功次数 / N_SIM
```
10000 次模拟在浏览器里瞬间完成（<50ms）。

**专武粗估（仅当勾选）：** 原神专武池保底机制不同（80抽硬保底、有定轨），v1 不精细建模，只给一句静态估算提示：「要专武的话，按平均约 80 抽，你的原石还差约 X 抽」。X = max(0, 期望总需求 - 当前抽数) 的粗略值。明确标注「专武为粗估，仅供参考」。

### 2.3 结论输出
模拟完成后展示：
- **决策灯**：🟢 抽 / 🟡 再等等 / 🔴 别抽
  - 阈值（v1 简单规则，可调）：到手概率 ≥ 70% → 🟢；40%~70% → 🟡；< 40% → 🔴
- **到手概率**：百分比 + 进度条
- **2~3 条人话理由**：基于输入动态生成，例如
  - 「80抽+已垫20+大保底，几乎稳出」
  - 「抽完大概率还剩约 10 抽」
  - 「要专武的话原石不够，差约 40 抽」
- **算法依据说明**：一行小字「依据官方公示概率 + 保底机制，蒙特卡洛模拟 1 万次」

### 2.4 分享功能
- 出结果后显示 **[生成分享图]** 按钮。
- 点击把「决策灯 + 到手概率 + 关键输入」渲染成一张**竖版图片**（适合手机截图发动态/群）。
- 图片底部带 **工具名 + 网址**，作为自然传播入口。
- 实现：用 `html2canvas`（CDN 引入）把结果卡片 DOM 转成图片下载，无需后端。

## 3. 技术方案

### 选型（小白友好 + Vercel 秒上线）
| 项 | 选择 | 理由 |
|---|---|---|
| 形态 | 纯前端静态单页 | 无审核、无账号、发链接即可测 |
| 技术栈 | 原生 HTML + CSS + JavaScript，**无构建步骤** | 小白门槛最低，一个文件夹直接部署 |
| 概率计算 | 浏览器内 JS 跑蒙特卡洛 | 无后端，10000 次模拟瞬时完成 |
| 分享图 | `html2canvas`（CDN） | 一个库搞定 DOM 转图，无后端 |
| 部署 | Vercel | 连 GitHub 自动部署，免费 HTTPS + 自定义域名 |

### 文件结构（极简）
```
/
├── index.html      # 页面结构 + 表单 + 结果区
├── style.css       # 样式（移动端优先，玩家多在手机看）
├── gacha.js        # 概率模拟 + 结论生成 + 分享图触发
└── docs/...         # 设计文档
```
无 package.json、无构建。Vercel 当作静态站点直接托管。

### 数据流
```
用户填表单 → 点「帮我决定」
  → gacha.js 读取输入
  → 蒙特卡洛模拟 10000 次 → 到手概率
  → 套阈值得决策灯 + 生成理由文案
  → 渲染结果卡片
点「生成分享图」→ html2canvas 截结果卡片 → 下载竖图
```

### 移动端优先
玩家主要在手机上看社区，页面按手机竖屏布局设计，桌面端居中单列即可。

## 4. 错误处理与边界
| 情况 | 处理 |
|---|---|
| 抽数为空/非数字/负数 | 表单校验，提示「请填写有效抽数」，不进入计算 |
| 已垫抽数 > 89 | 限制输入范围 0~89，超出则钳制并提示 |
| 命座目标 > 7 | 钳制到 7（满命） |
| 到手概率 0% 或 100% | 正常展示，理由文案对应「再多攒攒」或「稳了」 |
| html2canvas 加载失败 | 分享按钮降级为提示「截图保存即可」，不阻塞主功能 |

## 5. 测试与验收

### 概率正确性（关键，必须验证）
校验蒙特卡洛结果，方法：用**独立写的精确算法（动态规划/概率递推）**算同一场景，两者误差应在 ±2% 内。不靠手填的经验数字当标准（容易记错），而是两套独立实现互相对照。

可直接断言的边界基准：
| 场景 | 基准 | 验收 |
|---|---|---|
| 从0垫、大保底、90抽、目标1个限定 | ≈ 100%（第90抽硬保底必出，大保底必中限定） | ≈100% |
| 从0垫、50/50、0抽 | 0%（没抽数不可能到手） | 严格0% |
| 已垫89、50/50、1抽、目标1个 | 出5星必中（硬保底），但仅50%是限定 → ≈50% | ≈50% |
| 平均出一个限定所需抽数 | 社区公认 ≈ 93~94 抽 | 单独跑均值校验，落在区间内 |

其余场景（如「从0垫、50/50、90抽」）以 DP 精确解为准交叉验证，不预设固定百分比。

### 功能验收
